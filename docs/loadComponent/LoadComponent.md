# LoadComponent

`LoadComponent` gives an Entity a small, readable setup interface.

Instead of every map entry or spawner entry directly editing component fields, an Entity can expose a `load` payload. The `LoadComponent` receives that payload and maps it into normal component updates, executes, or switches.

Use `LoadComponent` for normal Entity setup. Use `Object.update` for direct one-off overrides.

## Two load styles

`LoadComponent` supports two payload styles.

### Array load

Array load is compact and positional.

```json
"load": [
    { "x": 358, "y": 436 },
    "PENCIL"
]
```

Array load uses numbered placeholders:

```json
"$value0"
"$value1"
"$value2"
```

### Named-object load

Named-object load is more readable for multi-field setup.

```json
"load": {
    "position": { "x": 358, "y": 436 },
    "itemType": "PENCIL"
}
```

Named-object load uses named placeholders:

```json
"$position"
"$itemType"
"$scale"
"$velocity"
```

## Important rule

The payload style must match the placeholders used by the Entity's `LoadComponent`.

This works:

```json
"load": {
    "position": { "x": 100, "y": 200 }
}
```

with:

```json
"options.position": "$position"
```

This works:

```json
"load": [
    { "x": 100, "y": 200 }
]
```

with:

```json
"options.position": "$value0"
```

This does not work:

```json
"load": {
    "position": { "x": 100, "y": 200 }
}
```

with:

```json
"options.position": "$value0"
```

because `$value0` only exists for array load.

The engine reports a clear unresolved-placeholder error when the styles do not match.

## Procedural values

Load payloads can use procedural value commands before they reach `LoadComponent`.

### `@range`

Creates a random number.

```json
{
    "@range": { "min": 0.07, "max": 0.10 }
}
```

Example:

```json
"load": {
    "scale": {
        "x": { "@range": { "min": 0.07, "max": 0.10 } },
        "y": { "@range": { "min": 0.07, "max": 0.10 } }
    }
}
```

### `@loadingRule`

Asks a component-provided loading rule for a value.

In `EntitySpawnerComponent`, this is commonly used to pass the spawner's generated position into the spawned Entity:

```json
"load": {
    "position": { "@loadingRule": { "this": "spawnPosition" } }
}
```

This means:

```text
Use the spawn position generated by the current spawner.
```

## When to use load vs update

Use `LoadComponent` when the Entity has a normal setup interface:

```json
"load": {
    "position": { "x": 100, "y": 200 },
    "scale": { "x": 0.1, "y": 0.1 }
}
```

Use `Object.update` when a map or spawner needs a direct one-off override:

```json
"update": {
    "component": {
        "#Motion": {
            "options.torque": { "@range": { "min": -30, "max": 30 } }
        }
    }
}
```

Recommended public pattern:

```text
Array load: compact, good for tiny shortcuts.
Named-object load: readable, preferred for multi-field setup.
Object.update: direct one-off overrides.
```

## Canonical debris example

`Debris` exposes a named-object load interface:

```json
{
    "type": "LoadComponent",
    "base": { "id": "Load" },
    "properties": {
        "editors": [
            {
                "update": {
                    "#Transform": {
                        "options.position": "$position",
                        "options.scale": "$scale",
                        "options.rotation.angle": "$rotation"
                    },
                    "#Motion": {
                        "options.velocity": "$velocity"
                    }
                }
            }
        ]
    }
}
```

Then a spawner can use readable named load values:

```json
"load": {
    "position": { "@loadingRule": { "this": "spawnPosition" } },
    "scale": {
        "x": { "@range": { "min": 0.07, "max": 0.10 } },
        "y": { "@range": { "min": 0.07, "max": 0.10 } }
    },
    "rotation": { "@range": { "min": 0, "max": 6.283185307179586 } },
    "velocity": {
        "x": { "@range": { "min": -120, "max": 120 } },
        "y": { "@range": { "min": -120, "max": 120 } }
    }
}
```
